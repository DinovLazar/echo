  - **Solid walls.** `Theme.swift` replaced the two-token wall gradient (`wallTop`/`wallBottom`) with one flat `wall` token per palette (Light `0x312D29` / Invert `0x2F2B25` — the gradient's visual midpoint, so the flat tone matches the old average). `BoardView.walls(cell:)` now fills `theme.wall` flat (was a `LinearGradient`); its doc comment is updated. Walls read as calm flat blocks — returning to the Plan §5 "walls are solid" intent (D-053).
  - **One reusable button style.** New `ECHO/Views/ControlButtonStyle.swift` — a single SwiftUI `ButtonStyle` every in-room action control wears, so Part 3's real controls reuse it rather than rebuild (D-054). A continuous-radius chip (`cornerRadius 11`): `theme.ink` @ 0.07 fill (→ 0.12 pressed) + a `theme.tileHairline` @ 0.6 hairline border, `theme.ink` footnote label, ~14/10 pt padding, a ≥44×44 pt tap target, scale-to-0.96 on press, and 0.35-opacity dimming when disabled. It reads the active palette via `@Environment(\.theme)` and the enabled state via `@Environment(\.isEnabled)`, so it adapts to Light/Invert and to `.disabled(...)`. Chips fill their container width (`maxWidth: .infinity`, `lineLimit(1)` + `minimumScaleFactor(0.85)` so a row divides into equal, evenly-spaced, single-line shares that fit any device). One **prominent** variant (`prominentFill`/`prominentLabel`) renders a solid fill + recoloured label — used only by the solved-room *Next* button.

The new `topHUD` is a `ZStack` with the level number (`roomIndex + 1`, 1–10) **truly centred** in its own layer, over an `HStack` of the Settings **gear** (leading, 44×44, unchanged `showingSettings = true` + `SettingsView` sheet) and the `turn N · echoes M/budget` readout (trailing). The gear, the readout, and the "Solved ✓" text are **gone from the bottom row** (gear + readout moved to the top HUD; "Solved ✓" replaced by the green Next).

When `state.hasWon`, the *Next* button uses the prominent variant — solid `theme.solvedGreen` fill + a `theme.paperTop` (paper-tone) label — as the "you may advance" signal; otherwise the normal chip. The green is **UI chrome only**: it appears on one button's state and **never on the board**, so D-041's two-accent board discipline (red/gold) holds (D-055, which partially supersedes D-041's no-third-colour clause for chrome only).
