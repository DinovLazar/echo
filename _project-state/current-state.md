## Summary (plain language)
- **Phase 3.04 (most recent change) — the groundwork for a real on-device feel/performance pass, plus two transition polish fixes.** Nothing here claims the game runs at 60fps or "feels right" — that can only be judged on Lazar's iPhone, which still hasn't happened. What this phase does:
  - **The two flagged transition glitches are fixed (D-064).** On "Next room," an opaque **paper cover** now veils the solved board **before** the room swaps, so the incoming board no longer flashes behind the outgoing win-overlay scrim. And the **win overlay** now reveals after a short **pre-delay** (≈0.45 s) so the solved board reads for a beat first. Both are named, commented "tune on device" first-guess constants in `RoomView` — the device pass adjusts them in one place. Presentation only; the engine, routes, collision, and win detection are untouched.
  - **A developer-only performance overlay (D-063).** A small frame-rate readout — FPS, instantaneous / rolling-average / worst frame time (ms), and a dropped-frame count over a window — mountable on **both** the campaign room and Echo Run (the stacked-shadow worst case) via `.debugPerformanceOverlay()`. It is gated **`#if DEBUG`** (compile-time), so it **can never appear in a release build**; it is not wired into the real Settings/HUD chrome, uses a neutral (non-`Theme`) pill so it doesn't touch the D-055 colour discipline, and is non-interactive so it can never eat a board swipe. The on-device session reads real numbers off it instead of eyeballing a fast board.
  - **A complete action→feedback audit (`docs/feel-audit.md`).** Every player-facing action in both modes mapped to its visual / audio / haptic channels, each marked **WIRED in code** (the call exists — verified by reading the source) vs **— none**, cross-referenced to the governing handover §/decision, with a top **"Gaps found"** section. ## Detail
- **`ECHO/Views/RoomView.swift`** — **Phase 3.04 (presentation edits, D-064).** Adds the two transition fixes: a `paperField` computed background reused as an opaque **advance cover**; `advancing` state + `advance(to:)` that raises that cover via `withAnimation(advanceCoverFade) { advancing = true } completion: { onAdvance(next) }` so the room swaps only once the outgoing board is hidden (no incoming-board flash behind the scrim); and a **win pre-delay** — the old `.onChange(of: state.hasWon)` is replaced by `.task(id: state.hasWon)` that marks the room solved immediately, holds `winRevealDelay` (≈0.45 s), then fades the win overlay in. Both `winRevealDelay` and `advanceCoverFade` are named, commented "tune on device" constants; `advance(to:)` is re-entry-guarded on `advancing` (a fast double-tap can't re-fire the swap). Mounts `.debugPerformanceOverlay()`. The HUD, controls, win-overlay content, and the input-lock guard are otherwise unchanged.
- **`ECHO/Views/EchoRunView.swift`** — **Phase 3.04.** Mounts `.debugPerformanceOverlay()` on the arcade board (the stacked-shadow worst case). Nothing else changed — the engine, the deferred-death dissolve, the game-over overlay are all as 3.03 shipped.
- **`ECHO/Views/PerformanceOverlay.swift`** — **new (Phase 3.04 / D-063).** The whole overlay is `#if DEBUG`. `View.debugPerformanceOverlay(_:)` is a `@ViewBuilder` that resolves to the modifier in debug and **`self` (a no-op) in release**, so the call sites compile in release with the overlay absent. In debug it overlays a top-trailing, non-interactive (`allowsHitTesting(false)`) neutral translucent pill showing FPS + instantaneous/avg/worst ms + dropped count. `FrameRateMonitor` (`@MainActor @Observable`) samples a `CADisplayLink` (iOS-only, behind `#if canImport(UIKit)`; off-iOS a no-op skeleton), records each frame into the pure `FrameWindow`, and republishes a `FrameStats` snapshot ~10×/s (so the readout's own redraws don't distort the measurement). `start()` resets the window (so a re-appear starts from fresh frames, not a stale spike).
- **`ECHO/Diagnostics/FrameStats.swift`** — **new (Phase 3.04 / D-063).** Foundation-only, `nonisolated`/`Sendable`. `FrameWindow` = a fixed-capacity rolling window of frame intervals (`record` ignores non-finite/non-positive; evicts oldest beyond capacity; `reset()` clears samples keeping capacity) with `averageSeconds`/`worstSeconds`/`fps` (guarded against 1/0) and `droppedCount(budgetSeconds:tolerance:)` (a hitch = interval > budget × tolerance, default 1.5×) + `summary` → the `FrameStats` snapshot. **Not** `#if DEBUG`-gated (harmless pure code, never instantiated in release) so it stays headlessly testable. Lives in `ECHO/Diagnostics/` — a dev instrument, kept out of `Models/`, but a pure value type in the same spirit (and so in the headless test module).
- **`ECHOTests/FrameStatsTests.swift`** — **new (Phase 3.04).** The `@MainActor` frame-math suite (+12 methods) described in the summary. Frame intervals chosen as exact binary fractions so equalities need no `accuracy:` tolerance.
- **`docs/feel-audit.md`** — **new (Phase 3.04).** The action→feedback checklist for the device session (both modes, three channels, WIRED-vs-none, spec-cross-referenced), the **Gaps found** section (G1–G7), and the **OWED on device** list. Documents gaps; invents none.
- **`ECHO/Models/*` / `EchoRunState.swift` / `Levels/*.json`** — the campaign engine + pure value types + the arcade engine + level data. **Byte-for-byte unchanged this phase.**
- **Everything else (`ECHOApp`, `AppRoute`, `ContentView`, `BoardView`, `BoardEffects`, `Theme`, `Motion`, `ControlButtonStyle`, `AudioManager`, `HapticsManager`, `SettingsStore`/`SettingsView`, `GuidanceController`/`GuidanceOverlay`, `Campaign`, `TitleView`/`MainMenuView`/`LevelSelectView`, `Assets.xcassets`, and the other test files)** — **unchanged this phase** (reused as-is).
