# Part 1 · Phase 05 · Code — Completion Report
**Date:** 2026-06-24 · **Outcome (one line):** Echoes are now lethal — after any committed step the player is checked against every echo, and touching one (land-on or cross-paths) dissolves present-you and restarts the current run while every folded echo survives and keeps replaying.

## 1. What shipped (plain language)
ECHO's one hard rule is now live. From this phase on, if your step lands you on a tile a past self occupies this turn — or you and a moving thing trade the same pair of adjacent tiles — present-you dissolves: you snap back to the start, the turn resets to 0, and the run you were walking is thrown away. Crucially, **all the echoes you've already folded survive a death and keep replaying** — only the live, in-progress run is discarded — so a death is "your fault, try that run again," not "start the room over." The rule lives entirely in the pure game model (so it's exactly, repeatably testable), and because the board is observable it reacts to a death for free (the player and every echo slide back to their starts). This phase's death is an immediate snap-back; the death "fizz" and freeze-frame are a later polish phase.

## 3. Decisions I made during this phase
- **Three prior 1.04 tests re-pathed to avoid the now-fatal echo retrace.** `testReplayMatchesLivePlayerPathTurnByTurn`, `testTwoEchoesOfDifferentLengthsStayLockedToSharedTurn`, and `testClearEchoesReturnsRoomToPristine` all advanced the live player by stepping `.up` — which exactly retraces the echo (whose recorded `sampleRun`/`runA` begins with `.up`), so under the new rule the player lands on its echo and dies, breaking the assertions (`turn` would be 0, not 1/2/3). *Decision:* change those live-play walks to non-colliding `.down` steps that exercise the **same** property (the echo stays locked to the shared turn / the room clears to pristine), and document the reason inline. *Why:* you cannot simultaneously have "collision exists" and "a test that walks the player through an echo expecting no death"; the test's purpose is preserved, only the tiles walked changed. *Rejected:* deleting the tests (loses coverage); special-casing them (dishonest). **Needs Decisions entry: NO** (test maintenance forced by the ratified D-018 behaviour; recorded here and in `current-state.md`).
- **Immediate snap-back, not sequenced, for the death animation.** The brief prefers showing the player reach the colliding tile before resetting, but allows an immediate snap-back if sequencing is fiddly. *Decision:* immediate snap-back. *Why:* Task 2/3 mandate that `move(_:)` performs the death restart, so the restart is synchronous — SwiftUI only ever observes the post-restart state, and making the touch visible would require pushing a timer/delay into the turn engine, compromising the determinism and testability that are hard design constraints (and unverifiable in this no-Xcode environment anyway). The visibly-sequenced touch + death "fizz" + freeze-frame are explicitly Phase 2.03. *Rejected:* a delayed/async restart in the model (breaks determinism); a view-level timer reading a death signal (adds async state for a frame that 2.03 will redo properly). **Needs Decisions entry: NO** (the brief pre-authorises the fallback; tracked in `current-state.md` Known issues).
- **Collision predicate is a single method over `echoes` with both clauses inline (not a generalised free function).** *Decision:* `playerCollides(previousPlayerCell:newPlayerCell:turn:)` loops `echoes`, deriving each echo's `t−1`/`t` tiles from `Echo.position`. *Why:* matches the brief's exact signature ("returns whether present-you collides with any echo") and stays the smallest correct thing; the land-on-OR-cross-paths shape is the general test Phase 1.06's hazards reuse, whether they share this array or a parallel one. *Rejected:* extracting a per-entity helper now (speculative generality before the hazard type exists). **Needs Decisions entry: NO** (D-018 already covers the land-on+cross-paths choice).
- **`playerCollides` is intentionally NOT turn-0-special-cased; turn-0 immunity is the engine's job.** *Decision:* the pure predicate is a plain land-on/cross test — at turn 0 an echo is on `start`, so calling it with the player on `start` correctly reports a land-on; immunity is achieved solely by `move(_:)` never calling it at turn 0. *Why:* keeps the predicate a clean, reusable geometric test (Phase 1.06 hazards may legitimately want to query any turn); the immunity is a property of when it's invoked. The turn-0 test asserts the **behaviour** (alive post-fold/post-death, no loop), not `predicate==false` at turn 0. *Rejected:* baking a `t == 0 → false` guard into the predicate (couples a render-stack concern into a geometry test). **Needs Decisions entry: NO.**
- **`move(_:)` returns `true` on a fatal step.** *Decision:* a step that lands fatally still returns `true` (a real tile move happened before it was discarded). *Why:* matches the existing "did a step commit" contract and is correct for the future move-tick feedback (you hear the tick, then dissolve); the death is observable via state. *Rejected:* returning `false` on death (would misreport that no step happened). **Needs Decisions entry: NO** (additive clarification; noted in the `move` doc-comment).
