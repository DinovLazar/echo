**Date:** 2026-06-26 · **Outcome (one line):** the grey-box board is now the real ECHO — every element drawn to the locked Phase 2.01 visual design in both Light and Invert palettes, with pieces that glide (player squash + soft-snap, echoes in lockstep, a heavier anticipating hazard) instead of snapping — engine, models, and tests untouched.

## 1. What shipped (plain language)
The board no longer looks like grey boxes. It now uses the real design: warm paper, near-black ink player, translucent grey echoes, a red diamond enemy with a soft glow, a gold goal ring, and proper walls/doors/switches — in two complete colour schemes (the normal Light one and a white-on-black Invert one you can flip with a debug button). Pieces slide smoothly between squares now: when you move, your square eases across with a little squash-and-stretch and a soft landing, your past-selves (echoes) glide along in perfect step, and the patrolling hazard moves a touch heavier with a small "wind-up" before it sets off. None of this changed how the game *works* — the turn engine, the level files, and the test suite were not touched; this phase only changed how the existing state is *drawn and animated*. The code type-checks cleanly against the real Swift 6.4 compiler + SwiftUI SDK, but because this Mac has only Command Line Tools (no Xcode), the actual on-device build, the ⌘U test run, and seeing the motion run at 60 fps are still owed on Lazar's machine.

## 3. Decisions I made during this phase
- **The single switch point is the `\.theme` SwiftUI environment value, set by `ContentView`.** Why: it is the idiomatic SwiftUI seam for a Settings phase to bind (2.06 swaps `ContentView`'s `@State themeMode` for a binding to a persisted preference) and lets `BoardView` read the palette with `@Environment(\.theme)`. Alternative rejected: a global `@Observable` settings object — heavier than this phase needs, and the brief explicitly says no toggle/persistence yet. Needs a Decisions entry: **NO** (within the brief's "environment value or a single observable property" guidance).
- **Added a throwaway *Invert* button to the debug bar.** Why: the DoD requires "Light and Invert both verified via the internal switch," and on-device verification without a flip would mean recompiling to change a default. The button is a debug control like *Next*, clearly **not** the real 2.06 Settings toggle (no persistence). Alternative rejected: flip a code default and rebuild per palette (worse for the owner's verification pass). Needs a Decisions entry: **NO** (consistent with D-017's throwaway debug bar).
- **The soft-snap is rendered as a keyframe scale overshoot, not a positional one.** Why: the 120 ms slide uses `curve.standard` (no overshoot) and the 40 ms soft-snap settle uses an overshoot; sequencing two animations on the *same* `.position` property is fragile, so the slide stays clean on `.position` and the ~6% overshoot lands as a scale settle in the tail — same "hair of weight" feel. Alternative rejected: a positional overshoot via a spring on `.position` (risks fighting the standard slide). Needs a Decisions entry: **NO** (an implementation choice within §6b's numbers; flagged for the on-device feel pass).
- **Retired the custom `Diamond: Shape`; the enemy is a rotated `RoundedRectangle`.** Why: the §2.6 enemy needs a small (3 pt) corner radius ("slightly sharp"); a rotated rounded square gives that cleanly with a built-in shape, and applying the squash/anticipation `scaleEffect` to the whole piece (in board axes) keeps the motion un-skewed by the internal rotation. This removes the Part 1 hotfix's subject (`Diamond`), but D-040's standing rule (any future custom `Shape` must be `nonisolated`) is unchanged. Alternative rejected: keep `Diamond` and hand-round its path (more code, error-prone). Needs a Decisions entry: **NO** (D-040 already covers the rule; this just means no custom shape exists now).
- **`commitMove` predicts the move outcome (read-only) to choose slide-vs-snap.** Why: the animation choice must wrap the mutation, but a death/fold must snap while a survived step slides — so the view reads the engine's public predicates before committing. The prediction only selects the *animation*; the engine still decides the move, so a mis-prediction could never change the outcome. Alternative rejected: mutate first then react (too late to wrap an animation). Needs a Decisions entry: **NO** (presentation-only; engine untouched).
- **Reconciled the echo-opacity discrepancy between the handover's §1b table and its §4 prose.** §1b lists echo fill/stroke 0.24/0.55; §4 says the Invert echo is 0.22/0.50. I treated §1b as the Light value and §4 as the Invert override (Light 0.24/0.55, Invert 0.22/0.50). Alternative rejected: force both modes to 0.24/0.55 (ignores §4's explicit invert values). Needs a Decisions entry: **NO** (a faithful reading of the handover; noted here for the orchestrator).
- **Invert player shadow read as a near-black drop shadow @ 0.40.** The §1b/§4 "ink-on-dark @ 0.40" is ambiguous; a light object lifts off near-black paper via a *darker* shadow, so I used black @ 0.40 (deeper than Light's ink @ 0.22). Alternative rejected: an off-white shadow (would read as a glow, not a lift). Needs a Decisions entry: **NO** (flagged for the on-device pass).
