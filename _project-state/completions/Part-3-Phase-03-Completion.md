# Part 3 · Phase 03 · Code — Navigation shell (Title, Main Menu, Level Select, win & game-over overlays)
**Date:** 2026-06-28 · **Outcome (one line):** ECHO is now a real app you move through — a Title screen, a Main Menu, a Level Select that remembers what you've solved, and proper win / game-over moments — with every throwaway debug button and shortcut deleted.

## 1. What shipped (plain language)
ECHO no longer boots into a debug board. It opens on a **Title** screen ("Echo" + the tap-to-begin motif), goes to a **Main Menu** (Campaign · Echo Run · Settings), and Campaign opens a **Level Select** — a grid of all 20 rooms that marks the ones you've solved (and saves that across relaunches) and highlights the next one to play. Picking a room plays it; reaching the exit shows a quiet **win panel** ("Next room" / "Level select", or "Campaign complete" on the last room). Echo Run now ends on a real **game-over panel** (your score, your best, Retry / Main menu). Every interim affordance from the old debug build — the `infinity` Echo-Run button, the debug `Next` room-cycle, the `Clear` button, and the temporary Echo-Run exit — is gone, and the in-room controls now correctly wait for an effect to finish before acting.

## 3. Decisions I made during this phase
1. **`Campaign` (a pure `nonisolated enum` in `ECHO/Models/`) is the single source of room order/identity.** The brief said "reuse the existing `roomIDs` list"; that list was a `private static` inside `ContentView`. I lifted it into `Campaign` (roomIDs + `next`/`number`/`firstUnsolved`/`contains`) so order/identity is one shared, **headlessly unit-testable** value (Level Select, RoomView, the win overlay, and the tests all read it). Alternative rejected: leave it in `ContentView` (untestable, and parallel numbering risk). This is the mechanism behind D-060/D-061 — no separate decision entry needed.
2. **The input lock is surfaced via one additive `inputLock: Binding<Bool>? = nil` on `BoardView`** (plus one `.onChange` reporting `fold != nil || death != nil`), and `RoomView` mirrors it into `@State inputLocked` and gates the controls. Alternative rejected: moving the control row *into* `BoardView` (a larger rewrite than "reuse `BoardView` as-is" allows). This is the established additive-parameter pattern (`audio`/`haptics`/`guidance` were all added this way) and touches no rendering/effect internal. Recorded in **D-059**.
3. **`ContentView` owns the campaign `GameState` (`roomState`), built fresh in `openRoom(_:)`; `RoomView` gets `.id(roomID)`.** This avoids a custom `@State` initializer in `RoomView` (which would break the CLT type-check harness, since `State(initialValue:)` doesn't match the `@StateShim` substitute) and keeps the per-room load flash-free entering from Level Select. Known minor artifact: the room→room "Next room" crossfade briefly shows the *new* board behind the *outgoing* (fading) win-overlay scrim, because `roomState` is shared during the ~200 ms fade — largely hidden by the outgoing scrim; flagged for device judgment.
4. **The in-room Settings gear was removed; Settings now lives only on the Main Menu** (Plan §3–§4 IA). The room's top-left affordance is a back chevron → Level Select. Consequence the owner should know: invert/sound/haptics can't be toggled mid-room — you go Menu → Settings. Alternative rejected: keep a gear in the room too (the brief's room spec lists no gear and reaches Settings from the menu).
5. **Win overlay reveals immediately on `hasWon` with a soft fade** (no delay). The winning 120 ms slide + solve flourish play under the fading scrim. A short pre-reveal delay is a cheap device-tuning candidate; not added (not a feel phase).
6. **Colour adjudication from the adversarial review (two reviewers disagreed):** the win-overlay closed exit-ring is **ink** (not gold), matching the Level-Select solved disc and keeping the green "Next room" as the overlay's single accent; the Echo Run "new best!" is **`solvedGreen`** (not gold), the D-055 chrome-success colour. One consistent rule across the app: **board goals = gold, chrome success = green, solved marks = monochrome.** (This also refines the carried-over 3.02 interim "new best!" gold.)
7. **`AppRoute` lives in `ECHO/App/` and the tests are written `AppRoute`-free**, so the documented headless test-module recipe (`Models/*.swift` + `SettingsStore` + `GuidanceController`) stays valid (`Campaign` rides `Models/*.swift`; `AppRoute` is app-only and only needs the app type-check). The no-gating invariant is tested via `Campaign` + `SettingsStore` instead.
8. **Partial-supersession handling of D-017 / D-026 / D-054 (see §4).**
